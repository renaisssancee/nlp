## Проект: AI HR Assistant — Resume ↔ Job Matching Agent

**Датасеты:**
- Jobs and Job Description (Kaggle): https://www.kaggle.com/datasets/kshitizregmi/jobs-and-job-description/data
- Resume Dataset (Kaggle notebook / dataset): https://www.kaggle.com/code/medelbachirkhellaf/resume-dataset

---

## 1. Контекст проекта

### 1.1 Бизнес-задача
- **Проблема:** соискатели не понимают, какие вакансии им реально подходят и чего не хватает в резюме; работодатели получают много нерелевантных откликов.
- **Почему решение имеет смысл:** keyword-based matching (ATS) плохо работает с синонимами, переносимыми навыками и контекстом опыта. Комбинация embeddings + RAG + LLM позволяет:
  - семантически сопоставлять резюме и вакансии,
  - извлекать и сравнивать навыки,
  - объяснять пользователю причины совпадения/несовпадения,
  - предлагать улучшения резюме и план “что подтянуть”.
- **Что считаем успехом (критерии):**
  - **Бизнес:** ≥70% положительных оценок рекомендаций (лайк/полезно), рост доли “релевантных рекомендаций” по ручной проверке.
  - **Технические:** Recall@5 ≥ 0.80, nDCG@5 ≥ 0.75 на тестовом наборе; P95 latency ≤ 2 сек (без учёта холодного старта).

### 1.2 Целевая аудитория и пользователи
- **Пользователи:** студенты, junior/middle кандидаты, люди в карьерном переходе (career switchers).
- **Сценарии использования:**
  1) загрузить резюме → получить топ вакансий + объяснение;
  2) вставить текст вакансии → получить “gap analysis” по резюме + рекомендации;
  3) (опционально) интерактивные уточнения: “какая роль интереснее?”, “какие ограничения по локации/зарплате?”
- **Интерфейсы:** Web (Streamlit) как MVP; Telegram-бот — опционально.
- **Нагрузка (MVP):** до 100–200 сессий/день, 5–10 одновременных запросов; цена ошибки — средняя (рекомендации).

### 1.3 Ограничения и допущения
- **Ограничения:**
  - не принимаем решения о найме, только рекомендации;
  - не гарантируем трудоустройство;
  - работаем с англоязычными резюме/вакансиями из открытых датасетов;
  - не храним и не обрабатываем реальные PII (в MVP только датасеты + тестовые резюме без персональных данных).
- **Допущения:**
  - данные Kaggle достаточно репрезентативны для демонстрации;
  - тексты вакансий содержат явные требования/скиллы;
  - пользовательские резюме можно привести к plain-text (PDF parsing вне MVP).

## 2. Архитектура решения

### 2.1 Общая схема
- **Пайплайн (MVP):**
  1) Пользователь загружает/вставляет резюме
  2) Resume Parser → нормализация текста
  3) Skill Extractor (LLM или правила) → список навыков + роль(и)
  4) Embedding резюме → запрос в Vector DB
  5) Retrieve top-K вакансий (K=20)
  6) ReRank top-K (cross-encoder) → top-N (N=5)
  7) LLM Agent: сравнение резюме с top-N + советы
  8) Ответ пользователю (структурированный + текстовый)

- **Примечание об актуальности данных:**
  В рамках MVP используется статический Kaggle-датасет вакансий.
  Цель проекта — демонстрация semantic matching и RAG-пайплайна,
  а не предоставление актуальных вакансий для трудоустройства.
  Архитектура retrieval-компонента спроектирована таким образом,
  что в продакшн-версии возможна замена источника данных
  на актуальные job API без изменения логики пайплайна.

- **Компоненты:**
  - Frontend: Streamlit (форма ввода резюме, вывод результатов)
  - Backend: FastAPI (оркестрация пайплайна)
  - Vector DB: Qdrant (или ChromaDB для локального MVP)
  - Embeddings: sentence-transformers (bi-encoder)
  - ReRanker: cross-encoder (например, multilingual/msmarco-подобный)
  - LLM: GPT-4o mini / Gemini (генерация объяснений и рекомендаций)
  - Logs/metrics: простое логирование + (опционально) Langfuse

## 3. Данные и качество знаний

### 3.1 Сбор и предобработка данных

**Источники данных:**

1. Job Descriptions Dataset  
   Формат: CSV  
   Поля:
   - ID (unique ID)
   - Job Title
   - Job Description

2. Resume Dataset  
   Формат: CSV  
   Поля:
   - ID
   - Resume_str (plain text)
   - Resume_html (HTML формат)
   - Category (label профессии)

---

**Предобработка вакансий (Job Dataset):**

- удаление HTML/спецсимволов (если встречаются)
- очистка дубликатов по (Job Title + Job Description)
- фильтрация коротких описаний (< 200 символов)
- нормализация пробелов
- объединение полей в единый текст для индексации:

  job_document =  
  "Title: {Job Title}  
   Description: {Job Description}"

---

**Предобработка резюме (Resume Dataset):**

- приоритет использования `Resume_str`
- `Resume_html` используется только если `Resume_str` пустой
- очистка HTML (если необходимо)
- нормализация текста
- извлечение навыков (через LLM extraction или rule-based approach)
- использование поля `Category` как:
  - ground truth для evaluation
  - дополнительная метка для анализа качества retrieval

---

**Метаданные для индексации вакансий:**

- job_id (ID)
- title
- inferred_category (опционально — через классификацию)
- source_dataset = "kaggle_jobs"

---

### 3.2 Векторизация и индексирование

**Embedding модель (MVP):**

- `sentence-transformers/all-MiniLM-L6-v2`  
  или  
- `intfloat/multilingual-e5-large`

Причины выбора:
- sentence-level embeddings
- оптимальный баланс скорость/качество
- поддержка semantic search

---

**Стратегия векторизации:**

Индексируем только вакансии.

Для каждой вакансии создаём embedding:

embedding(job_document)

При запросе:

embedding(resume_text)

---

**Метрика поиска:**
- cosine similarity

---

**Индекс:**
- Qdrant (HNSW)
- стартовые параметры:
  - M = 16
  - efConstruction = 200
  - efSearch подбирается эмпирически

---

**Reranking (второй этап):**

- bi-encoder → top-20 кандидатов
- cross-encoder → rerank top-20 → top-5

---

**Стратегия обновления:**

- инкрементальное добавление новых вакансий
- пересборка индекса при смене embedding модели
- версионирование индекса

---

### 3.3 Метрики качества знаний

#### 1. Retrieval-метрики

Так как в Resume Dataset есть `Category`, можно:

- проверять, совпадает ли категория резюме
  с категорией retrieved вакансий
- рассчитывать:

  - Recall@K
  - Precision@K
  - nDCG@K

---

#### 2. Консистентность

- отсутствие дубликатов
- корректная обработка пустых резюме
- корректный парсинг HTML

---

#### 3. Groundedness

- LLM рекомендации должны опираться только на:
  - текст резюме
  - текст вакансии
- ручная проверка 30–50 кейсов

---

#### 4. Санити-тесты

- резюме Data Scientist → top-5 содержит Data/ML роли
- резюме Marketing → top-5 содержит marketing роли
- random resume → не возвращает нерелевантные extreme категории

---

#### 5. Актуальность

Так как датасет статический (Kaggle):
- обновление не требуется в MVP
- для прод-версии возможна интеграция с job API

## 4. Модель и генерация

### 4.1 Выбор LLM и промптинг
- **LLM:** GPT-4o mini (или Gemini аналог). Достаточно reasoning для сравнения требований и опыта; хорошая стоимость/латентность.
- **Промпт-шаблон:**
  - вход: текст резюме + top-N вакансий (только ключевые фрагменты: title, requirements, skills);
  - инструкция: сравнить, выделить совпадающие/недостающие навыки, дать рекомендации, вернуть JSON.
- **Формат ответа:** строго JSON + короткое human-readable пояснение.
- **Ограничения:** ограничиваем контекст (top-5 после rerank), температуру 0.2–0.4.

### 4.2 Контроль качества ответов
- **Типы ошибок:** галлюцинации (“у вас есть опыт X”), оффтоп, слишком общие советы.
- **Механизмы:**
  - similarity threshold: если top-1 similarity < 0.35 → “недостаточно данных/уточните роль”;
  - prompt chaining: сначала извлечь skills из резюме (структурно), потом сравнивать;
  - (опционально) LLM-judge: проверка groundedness (“есть ли в резюме подтверждение навыка?”).
- **Метрики:**
  - доля ответов, соответствующих структуре JSON (валидность);
  - доля рекомендаций, подтверждённых контекстом (ручная оценка).

### 4.3 Обучение/дообучение
- **MVP:** без fine-tuning.
- **Потенциально:** fine-tuning на парах “резюме–вакансия–оценка соответствия” (если соберём разметку из пользовательского фидбэка).

## 5. UX / пользовательский опыт

### 5.1 Сценарии взаимодействия
- **Основные:**
  1) “Вставить резюме” → топ-5 вакансий + анализ;
  2) “Выбрать вакансию” → детальный skill-gap + советы, что поправить в CV;
  3) “Сгенерировать улучшение” → переписать bullet points под выбранную роль.
- **Исключительные:**
  - низкая уверенность retrieval → просьба уточнить роль/индустрию;
  - нет подходящих вакансий → просьба расширить описание навыков;
  - ошибка LLM API → сообщение + retry;
  - слишком длинный текст резюме → просим сократить/прикладываем авто-сжатие.

### 5.2 Диалоговая логика
- **MVP:** single-turn (один запрос → один ответ).
- **Память:**
  - short-term: в рамках сессии (последнее резюме, выбранная роль),
  - long-term: не храним по умолчанию.
- **Тон:** профессиональный, поддерживающий, без дискриминации.

### 5.3 Метрики UX
- Time-to-first-answer (TTFA), P95 latency.
- % пользователей, которые выбирают вакансию из топ-5.
- Like/Dislike на рекомендации.
- Retention (возвраты) — опционально.

## 6. Безопасность, соответствие и этика

---

Персональные данные:
- Резюме пользователя не хранится по умолчанию и обрабатывается только в рамках сессии
- Если пользователь вставляет PII (имя, телефон, email), предупреждаем и предлагаем удалить перед отправкой
- Логи сохраняются только в обезличенном виде и только с согласия пользователя

Защита от дискриминации:
- Бот не делает рекомендаций на основе защищённых признаков (пол, возраст, национальность, религия и т.д.)
- Промпт явно запрещает LLM упоминать или учитывать такие признаки при анализе

Фильтрация нежелательного контента:
- Бот работает строго в рамках тематики карьеры и резюме
- Запросы вне тематики обрабатываются fallback-ответом
- Галлюцинации LLM минимизируются через RAG: бот опирается только на текст резюме и вакансий из базы, а не на память модели

Прозрачность:
- При первом взаимодействии пользователь видит уведомление: *"Это AI-ассистент. Рекомендации носят информационный характер и могут содержать ошибки"*
- Бот не притворяется живым рекрутером
- Бот не принимает решений о найме, а только даёт рекомендации

Регуляторика:
- Датасеты (Kaggle) используются в соответствии с их публичными лицензиями
- LLM API (OpenAI / Gemini) используются согласно условиям использования провайдеров
- В случае работы с реальными пользователями необходимо соответствие GDPR (согласие на обработку данных, право на удаление)

## 7. План внедрения и эксплуатации

### 7.1 Этапы проекта

| Фаза | ДЗ | Что делаем |
|------|----|------------|
| Фаза 0 | ДЗ 1 | Дизайн-документ, выбор датасетов, планирование архитектуры |
| Фаза 1 | ДЗ 2 | Подготовка и очистка данных, индексация вакансий в векторную БД, базовый retrieval |
| Фаза 2 | ДЗ 3 | Reranking, LLM-агент, структурированный вывод (JSON), первичная оценка качества |
| Фаза 3 | ДЗ 4 | UI на Streamlit, логирование, улучшение промптов, финальная оценка, подготовка демо |

### 7.2 Поддержка и эксплуатация

Мониторинг:
- логирование ошибок и сбоев LLM API
- отслеживание latency (цель: P95 <= 2 сек)
- отслеживание частоты "low confidence" ответов (когда бот не уверен в рекомендации)

Обновление базы вакансий:
- в рамках учебного проекта используется статический датасет с Kaggle — обновление не предусмотрено

Ключевые метрики эксплуатации:
- uptime сервиса
- latency на запрос
- стоимость на один запрос (cost/request)

## 8. Риски и допущения
- **Риск:** слабое качество retrieval из-за “шумных” описаний вакансий.  
  **Mitigation:** очистка + rerank + hybrid search.
- **Риск:** LLM “придумывает” навыки пользователя.  
  **Mitigation:** prompt chaining + groundedness check.
- **Риск:** датасет не покрывает нужные профессии.  
  **Mitigation:** добавить второй источник вакансий или фильтровать на выбранные роли.
- **Риск:** высокая стоимость LLM при росте нагрузки.  
  **Mitigation:** кэширование, уменьшение контекста, дешёвые модели.

## 9. Бюджет и ресурсы
- **Люди:** 1 ML/NLP, 1 Backend, 1 Product/UX (может совмещаться).
- **Инфра:** CPU VM + Qdrant (docker).
- **LLM API:** ориентир 20–50$ / месяц для MVP (зависит от трафика).
- **ROI (для реального мира):** ускорение подбора релевантных вакансий и повышение качества резюме → меньше времени на поиски/отклики.
